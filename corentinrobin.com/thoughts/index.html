THOUGHTS ABOUT WEB DEVELOPMENT IN 2020

by Corentin ROBIN

version 27/11/2020

Definitions

We work inside a framework, and aside of a library.

Here we will use framework for both, to simplify.

Context

I’ve been working as a web developer since October 2016, and I’ve gathered some thoughts and some experiences during those years.

I worked as a full-stack developer during 2 and a half years, and then as a JavaScript developer, specialised in 2D graphics rendering (canvas).

I’ve been lucky enough to work for bosses that let me carte blanche.

But then I moved to Barcelona to start a new life, and things got a bit more complicated.

After around 10 interviews and about 4 technical tests, I understood something quite straight forward : front-end development in 2020 is focused on the method instead of the result.

Once I had a technical test asking me for a specific result, with specific frameworks.

I went straight vanilla as I was not an expert in those frameworks, I sent my solution.

I received the following answers : your solution works as it should, it’s good… but can you do the same with NewFancyFramework.js ?

Observations

Following this, few remarks :

  • if my solution is good, why should I redo it with NewFancyFramework.js ?
  • asking me to redo it with NewFancyFramework.js is for technical reasons or just for the sake of using NewFancyFramework.js ?
  • I have a technical background and I can read, don’t we think I’m able to adapt to a new method / tool ?

I found those following facts :

  • frameworks seem to be more a religion than an argumented technical need
  • a lot of companies choose the technologies even before starting to spec the actual project (a worker will choose his tools after knowing what he is suppose to build, not before)
  • we don’t hire developers, we hire NewFancyFramework.js developers (in 1953 Maserati hired Fangio as a driver, not as a Maserati driver ; Niel Armstrong was hired as a pilot, not as a spaceship pilot)

this framework religion is way less overwhelming back-end side : apart from Laravel or Symphony, no real mille-feuille of tools and libraries.

Asking if NewFancyFramework.js is really needed for a specific project will create a Holy War, where people will think that you are an old fashion guy, or a guy thinking different just for the sake of being a rebel, but they will never think that you are actually concerned by the value of using NewFancyFramework.js.

Examples of questionable frameworks

Webpack

I sent once my solution of a test (which was confirmed by the company itself as good), and I got the following size for the entire folder : 12.193 bytes for 10 ítems.

I was asked to use webpack and an ES2016 module way of doing, and for the exact same visual and functional result, I got the following size : 20.247.268 bytes for 6.531 ítems.

6500+ files ? Well, I can still use Prune.

But in order to use webpack, we need to install npm, and then use Prune to clean the node_modules folder, for an identical visual and functional result.

Tailwind CSS

In this example, we take Tailwind and analyse it, it order to determine how it will help us styling a page, compared to CSS.

if used 100%, Tailwind is both a library and a framework.

ConceptTailwindCSSEnd results
utility firstclass="bg-blue-600 text-white"style="background-color: var(--blue-600); color: white"pre-build classes, put inline in a class attribute : basically inline styles with different semantics
@apply directive.button { @apply bg-blue-600 text-white; }.button { background-color: var(--blue-600); color: white; }block containing pre-build Tailwind classes : basically CSS classes with different semantics
@tailwind directive@tailwind base;@import “base.css”;basically importing styles with different semantics
tailwind.config.jsmodule.exports = { [...] fontSize: { xs: '12px', sm: '14px' } [...] };:root { --font-size-xs: 12px; --font-size-sm: 14px; }configuration file to custom Tailwind pre-build classes : basically CSS variables with different semantics

The table above demonstrates that when used, Tailwind does the same than CSS, it's just that the spoken language is different.

inline styles is a list of CSS rules in an attribute style, Tailwind is a list of classes containing only one CSS rule each, in a class attribute.

If I say "Je mange une pomme.", "Yo como una manzana.", or "I eat an apple.", the language is different, but the idea expressed behind is exactly the same.

Both class="bg-blue-600 text-white" and style="background-color: var(--blue-600); color: white" mean the same thing: my element background is blue and its text color is white.

But you missed the fact that classes are already done for you, you will tell me. Well, I will finish this example with this open question : is it that long to build our own CSS set of classes ?

The framework business

At the beginning, the most used libraries where jQuery and Bootstrap, which never made any money with it, at a time where web development ecosystem was not as active as today on social networks.

Today, frameworks like Tailwind or Vue have 52k followers and 191k followers !

And they make business :

  • Tailwind UI
  • Tailwind Builder
  • Vue School
  • Vue Mastery

After knowing this, we can easily understand that the judgement can have a bias when choosing or not a framework.

Conclusion

In a more and more binary world, you are either anti or pro something, people need to put you in a category.

I’m not anti or pro frameworks, I’m just for having the right tool for the right task, and not for implementing NewFancyFramework.js because it's hype and has 87k followers on Twitter.

I think the following questions should appear more often in the numerous startup meetings:

  • about reusable components:
    • why defining a generic model of a website header if this header is unique and will never have more than 1 instance ?
    • how many pages we meet where we have several instances of the same UI model ?
    • who will show 10 todo-lists / notepads / checklists / etc. in the same page ?
  • example: products result page: all the UI components are passive, a simple PHP loop is sufficient to render the page. SPA you tell me ? Is it that UX-killer to do a quick page reload ?
  • other example: we know HTML, CSS, JS, PHP and MySQL. Is the framework selection + installation + learning + debug time will be below the standard languages development time ?

Why do I propose you will say. It's simple :

  • let's spec the project first, in the most exhaustive way as possible
  • let's check, for every part of the project, if we need tools, and if yes, what they will be
  • let's not be attracted by the hype of using X and Y framework
  • a bespoke framework can be built as well, in less time than you think (I worked for a company where I built the base code of a bespoke JS library for rendering 2D seating plans, with object zooming, panning etc. in 4 months, just with <canvas>)
